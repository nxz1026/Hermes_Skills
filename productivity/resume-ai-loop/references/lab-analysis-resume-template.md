# Lab-Analysis 简历章节模板

> 2026-06-22 实战产出。展示如何将一个 GitHub 项目的 README 深度信息转化为简历中的「代表成果」章节。

## 重写前（旧版简历）

```
医学检验数据分析Pipeline
独立设计并实现医学检验数据自动化分析Pipeline，覆盖从原始数据到临床建议输出的完整链路。
项目地址：https://github.com/nxz1026/Lab-Analysis
我的角色：独立开发者
核心工作：
多模态数据融合：集成检验指标（结构化）、MRI影像（非结构化）及PubMed文献（文本）三种数据源
AI Agent设计：设计多模型交叉验证机制（GLM-4V/DeepSeek/Qwen-VL）
...
```

**问题**：描述停留在「做了个 Pipeline」，没有体现 DSPy、证据打分、FHIR、评分卡、CI/CD 等深度特性。

## 重写后（新版）

见 SKILL.md Phase 3 的 Coder 任务模板中嵌入了完整新版内容。核心改进：

1. **架构特点前置**：11 步 Pipeline、DSPy 双轨、模块化设计
2. **关键能力具体化**：证据打分（5 维 + S/A/B/C）、FHIR R4、评分卡、反馈回路
3. **工程质量量化**：166 用例、CI 矩阵、跨平台、容错
4. **量化成果**：30min→3min，95% 一致性

## 改写原则

| 旧版写法 | 新版写法 | 原因 |
|---------|---------|------|
| "多模态数据融合" | "检验指标（CSV/JSON）+ MRI（DICOM）+ PubMed（文本）三源交叉印证" | 具体格式 > 抽象概念 |
| "AI Agent 设计" | "DSPy 程序化提示词优化，4 个模块双轨运行，BootstrapFewShot/MIPROv2" | 技术栈 > 泛泛而谈 |
| "端到端 Pipeline" | "11 步全自动 Pipeline，一键运行，中间产物可追溯" | 数字 > 形容词 |
| 无工程质量 | "166 个 pytest 用例，GitHub Actions CI（Python 3.10~3.12 矩阵）" | 证据 > 自称 |
| 无医疗标准 | "FHIR R4 标准输出，可直接对接 HIS" | 行业标准 > 通用描述 |

## 关键教训

- **读 GitHub README 再写简历**：简历里的「多模态数据融合」实际上还包含了 DSPy、证据打分、FHIR 等深层特性，不读 README 会漏掉关键卖点
- **用数字说话**：11 步、166 用例、5 维、3 场景、S/A/B/C——这些比「完整」「全面」「强大」有力得多
- **行业标准是加分项**：FHIR R4 是医疗 IT 的硬通货，写上比「可对接医疗系统」专业十倍
- **⚠️ 量化指标必须逐项核对源材料**（2026-06-22 Auditor 发现）：
  - 「解读一致性95%」→ DSPY_INTEGRATION.md 中 95% 是**格式合规性**（format compliance），解读一致性实际为 90%。两个指标不可混用
  - 「处理时间从30分钟缩短至3分钟」→ 源代码 README + docs 中**无此 benchmark**。如为实测值须标注「自测」，否则删除
  - **原则**：简历中的每个数字都必须在 GitHub README/docs 中找到对应出处，找不到出处的数字一律不写
